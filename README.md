# Climber Tier Chart

A power ranking of the world's hardest outdoor climbers, across **sport**, **bouldering**, and **overall**, with an **All-Time / Current** toggle. Static site, hosted on GitHub Pages. The data is scraped monthly from [climbing-history.org](https://climbing-history.org) by a GitHub Action — no server, no local cron, no computer-on requirement.

## How it ranks

Each climber gets a score per discipline:

```
score = sum(top-10 hardest qualifying sends)
      + 0.5 × sum(top-5 hardest qualifying flashes/onsights)
```

Each grade is worth points on an exponential scale (3<sup>×</sup> per step):

| Sport | Pts | Boulder | Pts |
|---|---|---|---|
| 9b   | 1   | V16 | 1 |
| 9b+  | 3   | V17 | 3 |
| 9c   | 9   | V18 | 9 |
| 9c+  | 27  | V19 | 27 |

So one 9c is worth as much as nine 9b's; one V18 = nine V16s. The exponential weight is what keeps deep résumés from drowning out hardness, and vice versa.

For **Overall**, a climber's sport and boulder scores are summed.

## Qualifying thresholds

Sends below these grades don't count (toward score, ranking, or display):

|                  | Sport | Boulder |
|------------------|-------|---------|
| Redpoint floor   | 9b    | V16 / 8C+ |
| Flash/onsight floor | 8c+ | V14 / 8B+ |

All thresholds and weights live in [`config.yaml`](config.yaml).

## How "Current" works

`Current` ranks using only sends from the last 3 years (configurable via `config.current_years`).

## Repo layout

```
config.yaml                # thresholds, weights, leaderboard size
climbers.yaml              # generated by fetch_data.py — DO NOT edit by hand
data.json                  # generated by build.py — what the site loads
index.html                 # single-file frontend
scripts/fetch_data.py      # climbing-history.org → climbers.yaml
scripts/build.py           # climbers.yaml + config.yaml → data.json
.github/workflows/update.yml  # monthly cron + on-push rebuild
```

## Data source

Six canonical lists from climbing-history.org are scraped (sends only, no attempts):

- `/list/5`  — sport ascents 9b+
- `/list/13` — sport ascents 9a+ by women
- `/list/14` — boulder ascents 8C+
- `/list/20` — boulder flashes 8B+
- `/list/37` — sport onsights 8c+
- `/list/42` — sport onsights 8b+ by women

The data is reset on every fetch — no manual edits to `climbers.yaml` will survive the next run. To correct a bad ascent, fix it on climbing-history.org and the next monthly run picks it up. To change ranking behavior, edit `config.yaml`.

## Local development

```bash
pip install -r requirements.txt
python scripts/fetch_data.py       # rebuild climbers.yaml from climbing-history.org
python scripts/build.py            # rebuild data.json from yaml + config
python -m http.server 8000         # open http://localhost:8000
```

## Deploying on GitHub

1. Push to GitHub.
2. **Settings → Pages → Build and deployment → Source: GitHub Actions.**
3. The workflow runs on push, on the 1st of every month at 06:00 UTC, and on demand from the Actions tab.
